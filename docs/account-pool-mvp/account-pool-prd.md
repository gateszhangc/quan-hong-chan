## 2. 背景与目标

### 2.1 背景
EasyClaw 平台需要为用户提供 OpenClaw 部署服务。在 MVP 阶段，平台采用**管理员预置账号池**的模式：
- 管理员提前准备一批 OpenAI Codex 账号
- 用户首次部署时自动从账号池分配一个未使用的账号
- 分配后账号与用户 **1:1 永久绑定**
- 后续部署复用已绑定的账号

### 2.2 目标
- 实现账号池的自动化分配与绑定机制
- 确保用户部署时始终有可用的 OpenAI 账号
- 支持 OAuth Token 的自动续期
- 保证账号数据的安全存储

### 2.3 非目标
- 不支持用户自带账号（BYO）
- 不支持账号共享/轮换使用
- 不支持多账号绑定（一个用户只能绑定一个账号）

## 3. 用户故事

### 3.1 管理员视角

**US-001: 批量导入账号**
> 作为管理员，我希望通过脚本批量导入 OpenAI 账号的 OAuth Token，以便快速初始化账号池。

**US-002: 查看账号池状态**
> 作为管理员，我希望查看账号池的使用情况（总数、已绑定、可用、失效），以便及时补充账号。

**US-003: 账号失效处理**
> 作为管理员，我希望当账号被封或 Token 失效时自动标记，以便清理和替换。

### 3.2 用户视角

**US-101: 首次部署自动绑定**
> 作为用户，当我首次点击"部署"时，系统自动为我分配一个 OpenAI 账号，我无需关心账号来源。

**US-102: 再次部署复用账号**
> 作为用户，当我再次部署时，系统使用之前绑定的同一个账号，我的配置和数据保持一致。

**US-103: 部署失败提示**
> 作为用户，当账号池耗尽或账号失效时，我看到清晰的错误提示，知道需要联系管理员或稍后再试。

## 5. 非功能需求

### 5.1 性能
- 账号分配操作 < 100ms
- Token 刷新操作 < 2s
- 不影响现有部署流程的耗时

### 5.2 安全
- 所有 Token 必须加密存储
- 加密密钥通过环境变量管理
- 数据库不存储明文 Token
- 容器内 Token 通过 volume 挂载，不暴露在环境变量

### 5.3 可用性
- 账号池耗尽时优雅降级（提示用户而非崩溃）
- Token 刷新失败时标记账号失效，不影响其他账号
- 支持并发部署（多个用户同时部署不冲突）

### 5.4 可维护性
- 提供账号导入脚本
- 提供账号状态查询接口（管理员）
- 支持手动标记账号失效

## 6. 验收标准

### 6.1 功能验收

| 验收项 | 标准 |
|--------|------|
| AC-001 | 管理员可以通过脚本成功导入 10 个账号到账号池 |
| AC-002 | 用户 A 首次部署，成功分配账号 X，容器正常运行 |
| AC-003 | 用户 A 再次部署，复用同一个账号 X，容器正常运行 |
| AC-004 | 用户 B 首次部署，分配账号 Y（与用户 A 不同） |
| AC-005 | Token 过期后，系统自动刷新，部署成功 |
| AC-006 | Token 刷新失败，账号标记为失效，返回友好错误 |
| AC-007 | 账号池耗尽时，返回"暂无可用账号"提示 |
| AC-008 | 并发情况下，10 个用户同时部署，各自分配到不同账号 |

### 6.2 安全验收

| 验收项 | 标准 |
|--------|------|
| AC-201 | 数据库中 Token 为加密存储，无法直接读取 |
| AC-202 | 容器内通过文件挂载获取 Token，环境变量不包含 Token |
| AC-203 | 无加密密钥时，系统拒绝启动 |

## 7. 风险与限制

### 7.1 合规风险
- OpenAI ToS 可能禁止账号共享/代理使用
- **缓解措施**：MVP 阶段小范围测试，后续迁移到 BYO 模式

### 7.2 账号被封风险
- 平台批量使用的账号可能被 OpenAI 检测并封禁
- **缓解措施**：
  - 限制单个账号的调用频率
  - 账号失效后自动标记
  - 准备充足的账号池冗余

### 7.3 技术限制
- 每个用户只能绑定一个账号（不支持多账号切换）
- 不支持用户主动解绑/换绑（需管理员介入）
