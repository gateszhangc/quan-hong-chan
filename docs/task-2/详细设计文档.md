# EasyClaw「一键部署 OpenClaw」Design Doc（DD v3）

- 版本：v3
- 更新：2026-02-08
- 面向读者：后端 / 前端 / 运维

> 本设计文档采用 `design-doc` skill 的模板结构：Context → Goals/Non-goals → Behavior → API/State → Error/UX → Testing → Acceptance。

---

Context and motivation
----------------------

EasyClaw 的目标是把 OpenClaw 的部署与通道接入做成“一键可用”。
当前 repo 已有可运行的实现（Express + Dockerode + Next.js rewrites），但需要把真实实现与边界写清楚，确保：
- 部署是异步的（HTTP 不阻塞）
- 成功是“真的可用”（gateway + telegram provider ready）
- Codex OAuth 模式可运行（依赖账号池导入与分配）

Goals:
- 用户提交 Telegram Bot Token 后，系统自动拉起 OpenClaw 容器并配置完成
- 前端可以轮询查询部署状态（provisioning/running/failed）
- 以 `openai-codex/gpt-5.2` 为默认模型，部署无需 `OPENAI_API_KEY`
- 凭证加密存储，日志脱敏

Non-goals for first implementation (v1):
- 不做计费、组织/团队、多实例编排
- 不做多通道（Discord/WhatsApp）实际接入
- 不做完整控制台（Cron/Skills/日志全文搜索）

---

Implementation considerations
-----------------------------

关键约束与原则：
- **HTTP 不做重活**：`POST /api/deploy` 只创建记录并触发异步部署
- **可用性优先**：仅当 gateway + telegram provider ready 才标记 running
- **最小权限与脱敏**：token 必须加密；日志不含 token
- **并发安全**：账号池分配使用 `FOR UPDATE SKIP LOCKED`
- **可恢复**：容器 `RestartPolicy=unless-stopped`

与现有代码对齐的实现要点（真实路径）：
- 后端入口：`backend/src/index.ts`
- Deploy API：`backend/src/routes/deploy.ts`
- Docker 编排：`backend/src/services/docker.ts`
- 加密：`backend/src/utils/crypto.ts`
- 账号池：`backend/src/services/account-pool.ts`
- Next.js rewrites：`next.config.mjs`（`BACKEND_BASE_URL`）

High-level behavior
-------------------

从用户视角的端到端行为：

1) 用户在前端选择 model/channel 并输入 Telegram bot token
2) 前端调用 `POST /api/deploy`
3) 后端：
   - 校验 payload
   - 生成 deploymentId
   - 加密保存 telegram token
   - 立即返回 deploymentId
   - 在后台异步执行部署（Docker pull → run container → exec openclaw config → ready check）
4) 前端每 2s 轮询 `GET /api/deploy/:id`
5) 后端返回：
   - `running`：部署可用
   - `failed`：失败并返回 `error_message`

Happy path 下，“running”意味着：
- 容器处于 Running 状态
- OpenClaw gateway 已开始监听（日志包含 `listening on ws://`）
- Telegram provider 已启动（日志包含 `[telegram]` + `starting provider`）

API design
----------

### POST /api/deploy
- 实现：`backend/src/routes/deploy.ts`

Request:
```json
{
  "telegram_token": "123456:ABC...",
  "model": "gpt-5-2"
}
```

Response:
```json
{
  "deployment_id": "<uuid>",
  "status": "provisioning"
}
```

行为细则：
- 若 `AUTH_DISABLED=true`，后端会将 userId 固定为 `test-user`
- `model` 可选；未传时默认走后端默认模型映射（当前实现默认 `openai-codex/gpt-5.2`）

### GET /api/deploy/:id
Response:
```json
{
  "status": "provisioning|running|failed|stopped",
  "error_message": "..."
}
```

> 现状：GET 接口未强制 requireAuth。生产建议校验 deployment 属于当前 user。

### Admin: 账号池
实现：`backend/src/routes/admin-accounts.ts`

- `POST /api/admin/accounts/import`：导入 `~/.codex/auth.json`
- `GET /api/admin/accounts`：列表 + stats
- `DELETE /api/admin/accounts/:id`：删除（可 force）
- `POST /api/admin/accounts/:id/unbind`：解绑

**安全要求**：生产必须给 admin 路由加保护（当前实现未加）。

State model
-----------

### deployments.status
- `provisioning`：部署中
- `running`：可用（ready check 通过）
- `failed`：失败
- `stopped`：可选（解绑/停止）

### 账号池状态
- `account_pool.is_bound`：是否已绑定 user
- `account_pool.is_active`：是否可用（refresh 失败会标 false）

Data model and storage
----------------------

数据库以 `src/db/migrations/` 为准（优先于 `backend/sql/init.sql`）。核心表：

### deployments
- `id` uuid
- `user_id` varchar
- `account_id` uuid（可选，关联 account_pool）
- `status` varchar
- `telegram_token_encrypted` text
- `error_message` text
- `stopped_at` timestamptz
- `stop_reason` varchar
- timestamps

### account_pool
- `access_token_encrypted`, `refresh_token_encrypted`, `expires_at`
- `account_id`（OpenAI account id，unique）
- `is_bound`, `bound_user_id`, `is_active`, `failure_count`, `last_used_at`

### account_unbind_logs
- 解绑审计

Secrets 加密
- 机制：AES-256-GCM
- Key：`ENCRYPTION_KEY`（hex64 或 base64）
- 实现：`backend/src/utils/crypto.ts`

Docker / OpenClaw container behavior
------------------------------------

实现：`backend/src/services/docker.ts`

### 镜像与容器
- Image：默认 `fourplayers/openclaw:2026.2.9`（`OPENCLAW_IMAGE` 可覆盖）
- 容器名：`openclaw-<deploymentId>`
- RestartPolicy：`unless-stopped`

### 注入环境变量（容器内）
- `TELEGRAM_BOT_TOKEN`（必需）
- `OPENCLAW_MODEL`（例如 `openai-codex/gpt-5.2`）
- `OPENCLAW_GATEWAY_TOKEN`（gateway 鉴权）
- `OPENCLAW_GATEWAY_MODE`（默认 local）
- `OPENCLAW_ALLOW_UNCONFIGURED=true`
- `OPENCLAW_SKIP_ONBOARD=true`

### 容器内配置步骤（关键）
1) `mkdir -p /home/node/.openclaw/agents/main/agent`
2) 写入 `auth-profiles.json`（如果分配到账号池）：
   - profileId：`openai-codex:codex-cli`
   - provider：`openai-codex`
3) `openclaw config set gateway.mode <local|...>`
4) Telegram 配置：
   - `channels.telegram.allowFrom=["*"]`
   - `channels.telegram.dmPolicy=open`
   - 删除 pairing 文件：`~/.openclaw/credentials/telegram-pairing.json`
5) `openclaw models set <model>`
6) `openclaw models status --json` 校验缺失凭证
7) 重启 gateway（pkill 进程，supervisord 拉起）
8) 等待 Ready：gateway + telegram provider 都启动

Error handling and UX
---------------------

错误分类与用户提示建议：

1) 用户输入错误
- Telegram token 无效
  - 期望提示："Telegram token 无效，请检查是否复制完整"
  - 建议：部署前 `getMe` 预校验（减少等待）

2) 资源/依赖错误
- Docker socket 不可用 / 无权限
- 镜像拉取失败
- 容器启动后立即退出

3) 外部账号错误（Codex OAuth）
- 账号池无可用账号：`NO_AVAILABLE_ACCOUNT`
- refresh token 失效：标记账号 inactive，并提示管理员补充账号

前端 UX（当前实现为轮询）：
- Deploying…
- running：显示下一步
- failed：显示 error_message + 建议重试

---

Update cadence / Lifecycle
--------------------------

- 部署创建后立刻异步执行；前端轮询 2s
- running 后容器常驻，Docker 负责重启（unless-stopped）
- 账号池 token 接近过期（<5min）时，后端在使用前刷新

Future-proofing
---------------

后续扩展点：
- 多通道：Discord/WhatsApp（需要通道凭证与 OpenClaw 配置写入）
- 多模型：把 `modelMap` 配置化（DB/配置文件），并在 UI 上按“可用性”动态展示
- 部署队列：把 setImmediate 替换为 BullMQ/Cloud Tasks，支持并发与重试
- 进度事件：增加 `deployment_events` 表 + SSE 推送
- 安全收敛：Telegram allowFrom 从 `*` 改为 allowlist/pairing

Implementation outline
----------------------

1) API 层
- 强化 `POST /api/deploy` 参数校验（token 格式、model allowlist）
- 增加 token 预校验（Telegram getMe）

2) 部署执行
- 将部署步骤拆分并记录 progress（pull/config/ready）
- Ready check 超时信息更可读（回传末尾日志片段，脱敏）

3) 账号池
- Admin import 增加鉴权保护
- 对 refresh 失败增加 failure_count 与告警策略（后续）

4) 安全与权限
- `GET /api/deploy/:id` 增加 ownership 校验
- /api/deploy rate limit

Testing approach
----------------

- Unit
  - crypto 加解密（ENCRYPTION_KEY 格式覆盖）
  - authJson 解析（admin import）
  - model 映射逻辑

- Integration
  - deployments 表写入/更新
  - account_pool 分配并发（SKIP LOCKED）

- Manual / E2E
  - 用真实 Telegram bot token 部署一次
  - 验证 running 后 bot 能回复
  - 验证失败场景：无可用账号、Docker socket 不可用（可用错误注入）

Acceptance criteria
-------------------

- Given 有效 Telegram token + 账号池有可用账号
  - When 调用 `POST /api/deploy`
  - Then 2 秒内返回 deployment_id，最终状态变为 running
  - And bot 在 Telegram 可响应一条消息

- Given Telegram token 无效
  - When 调用 `POST /api/deploy`
  - Then 返回 4xx（或快速 failed），并给出可读错误信息

- Given 账号池无可用账号
  - When 用户发起部署
  - Then 部署失败，error_message 明确指出“无可用账号”

- Admin 可以导入 `~/.codex/auth.json` 并在列表中看到账号；刷新失败会将账号标为 inactive
