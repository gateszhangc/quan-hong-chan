# EasyClaw「一键部署 OpenClaw」— 需求文档（PRD v3）

- 版本：v3
- 更新：2026-02-08
- 面向读者：产品 / 前后端 / 运维

> 本 PRD 按“先 Why / Why now → 再 Success → 再 What（范围）”的结构来写，确保工程能直接开干。

---

## 1. 背景（Context）与问题（Problem）

OpenClaw/Clawdbot 是“常驻代理 + 工具 + 定时任务”的系统。
对多数目标用户来说，**价值**很明确（提醒/日报/自动化/助手），但**落地门槛**集中在部署链路：

- 需要服务器、Docker、网络与端口
- 需要通道接入（Telegram/Discord 等）
- 需要模型凭证（API Key 或 OAuth）
- 需要把配置写对（openclaw.json/.env）

用户真正想要的不是“懂部署”，而是“立刻可用”。

### Why now（为什么现在做）
- OpenClaw 的能力可扩展，但若部署成本高，转化率会被卡死在第一步
- 先把“一键跑起来”做成标准化路径，后续再加控制台/计费/多通道才有意义

---

## 2. 目标（Goals）与成功定义（Success Criteria）

### 2.1 核心目标
1) **TTD（Time to Deploy）**：用户点击 Deploy → 在 Telegram 收到 bot 回复，尽量 < 1 分钟（可接受 < 3 分钟）。
2) **部署成功率**：成功 / 总部署请求 ≥ 80%（MVP 目标；随稳定性逐步提高）。
3) **失败可解释**：失败必须能给出用户可理解的原因与下一步。

### 2.2 可量化指标（建议埋点）
- CVR：Landing → 点击 Deploy → running
- TTD 分布：P50 / P90
- 失败原因 Top N：Token 无效 / 无可用账号 / 拉镜像失败 / 配置失败 / Ready 超时
- 7 日留存：部署成功用户 7 日内再次发消息并得到回复

## 3. 用户画像（Personas）与 JTBD

1) **个人用户/独立开发者**
- JTBD：我想要一个能发日报和提醒的 bot，但我不想研究服务器。

2) **小团队运营/客服**
- JTBD：我想把 bot 接到 Telegram，能做自动回复/信息汇总。

3) **技术负责人（评估者）**
- JTBD：我需要一个可控、可扩展、凭证不乱飞的部署方案。

## 4. 范围（Scope）

### 4.1 MVP（必须交付）
- **通道**：只支持 Telegram Bot（其他通道 UI 可展示但必须禁用或标 Coming soon）。
- **默认模型**：`openai-codex/gpt-5.2`（Codex OAuth）。
  - 关键点：不依赖 `OPENAI_API_KEY`，依赖“账号池（account_pool）”分配 OAuth token。
- **一键部署**：用户输入 Telegram Bot Token → 点击 Deploy → 状态可视化（provisioning/running/failed）。
- **账号池管理**：管理员可导入 Codex CLI 的 `~/.codex/auth.json` 到账号池；用户部署时自动绑定/刷新/失效处理。
- **安全底线**：敏感 token 必须加密存储；日志不得出现明文。

### 4.2 明确非目标（MVP 不做）
- 计费（订阅/credits）
- 多实例管理（一个用户 N 个实例）
- 多通道并存（同一实例 Telegram+Discord）
- 完整控制台（Cron、技能市场、日志全文检索）

---

## 5. 核心用户路径（User Journey）

### 5.1 首次部署（Happy path）
1. 进入 Landing
2. 选择 Model（MVP：仅 GPT-5.2 可选）
3. 选择 Channel（MVP：仅 Telegram 可选）
4. 弹窗指引用户去 @BotFather 创建 bot，粘贴 token
5. 点击 Deploy
6. 显示状态：Deploying…（轮询）
7. 成功：显示“下一步”
   - 去 Telegram 找到 bot
   - /start
   - 发一句“hello”验证可用

### 5.2 失败修复（必须覆盖）
- Token 无效：应尽量在部署前预校验（Telegram `getMe`），直接提示，不让用户等待。
- 无可用账号：提示“当前无可用账号，请联系管理员导入账号”。
- 拉镜像失败/启动失败：提示“稍后重试”，并提供部署 ID 供定位。

## 6. 功能需求（Functional Requirements）

> 优先级标记：MUST / SHOULD / COULD

### 6.1 Landing：模型与通道选择
- MUST：清晰展示可选项；对不可用项禁用并标注 Coming soon。
- SHOULD：默认预选 `GPT-5.2` + `Telegram`，减少用户操作。
- MUST：UI 选择结果必须参与部署请求（model/channel）。

### 6.2 Telegram Token 引导与校验
- MUST：引导流程（@BotFather、/newbot、复制 token）。
- MUST：前端基础校验：非空 + 包含 `:`。
- SHOULD：后端部署前预校验：`https://api.telegram.org/bot<TOKEN>/getMe`。
- MUST：token 作为敏感信息：
  - 后端加密存储
  - 前端与日志不得打印明文

### 6.3 部署 API
- MUST：`POST /api/deploy` 创建部署，2 秒内返回 `deployment_id`。
- MUST：部署执行异步（不阻塞 HTTP）。
- MUST：`GET /api/deploy/:id` 返回 status + error_message。
- SHOULD：状态细分 progress step（pull/config/ready），便于 UI 更友好。

### 6.4 Ready 判定（可用性）
- MUST：不能只用“容器 Running”判定成功。
- MUST：至少满足：gateway ready + telegram provider ready（以日志/健康探针为准）。

### 6.5 账号池（Codex OAuth）
- MUST：管理员导入 `~/.codex/auth.json` → `account_pool`。
- MUST：部署时对每个 user 分配或复用绑定账号（并发安全：SKIP LOCKED）。
- MUST：token 自动刷新；刷新失败要下线账号（is_active=false）。
- SHOULD：账号池统计接口（total/bound/available/inactive）。

### 6.6 鉴权与权限
- MVP 允许：`AUTH_DISABLED=true`（便于本地调试）。
- SHOULD：生产启用鉴权（Bearer token → userId）。
- MUST：管理员接口必须受保护（最少 admin key / 仅内网）。

## 7. 非功能需求（NFR）

- 安全：
  - MUST：敏感信息加密（AES-256-GCM）
  - MUST：错误信息脱敏
  - SHOULD：/api/deploy rate limit
- 稳定：
  - SHOULD：容器 RestartPolicy=unless-stopped
- 性能：
  - SHOULD：Landing LCP 尽量 < 2.5s（图片懒加载、减少大脚本）
- 可观测：
  - SHOULD：以 deployment_id 为主线记录关键事件

---

## 8. 验收标准（Acceptance Criteria）

1) MVP 只允许 Telegram 部署：Discord/WhatsApp 不可点击或明确 Coming soon。
2) MVP 默认模型为 `openai-codex/gpt-5.2`，部署成功后 bot 可响应。
3) `POST /api/deploy` 返回 `deployment_id`，前端轮询能反映状态变化。
4) 失败场景可解释：Token 无效/无可用账号/镜像拉取失败等至少覆盖 3 类。
5) 管理员能导入至少 1 个账号到账号池，并能看到账号列表/统计。

## 9. 开放问题（Open Questions）

1) 生产环境要不要允许“每用户多个实例”？（影响资源与 UI）
2) Telegram 安全策略：allowFrom="*" 是否要收敛为 allowlist / pairing？
3) BACKEND_BASE_URL 的网络策略：直连公网 IP 还是通过网关/内网？
4) 账号池的运营：账号来源与失效处理流程如何制定？
